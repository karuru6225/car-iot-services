# OBD2 / Honda N-VAN 調査メモ

## 概要

OBD2（On-Board Diagnostics 2）は車両の自己診断システムの標準規格。
1996年以降の北米車、2001年以降のガソリン欧州車、2004年以降のディーゼル欧州車で義務化。
日本では2008年モデル以降は CAN が必須。

Honda N-VAN (JJ1/JJ2, 2018年7月〜) はCAN通信を使用する。

---

## 物理層・プロトコル

### OBD2 コネクタ（16ピン Dサブ型）

```
ピン 6  : CAN H
ピン 14 : CAN L
ピン 16 : バッテリー電源（12V系）
ピン 4/5: GND
```

### N-VAN の CAN バス構成

みんカラ「トレ猫」さんによるリバースエンジニアリング結果:

| バス | 速度 | ID フォーマット | トランシーバ | 用途 |
|------|------|----------------|------------|------|
| **F-CAN** | 500 kbps | 標準 11bit | TJA1051 | エンジン・AT系 |
| **B-CAN** | 125 kbps | 拡張 29bit | TJA1042 | ボディ系（ドア・ライト等） |

OBD2 コネクタからアクセスできるのは基本的に **F-CAN**。

---

## OBD2 メッセージ構造（CAN ベース）

### CAN フレーム

```
リクエスト: ID=0x7DF（全ECUへのブロードキャスト）
レスポンス: ID=0x7E8（エンジンECU）、0x7E9 以降（他ECU）
```

### データフォーマット（8バイト）

```
[len][mode][PID][data...][padding]
```

**リクエスト例**（車速）:
```
02 01 0D 00 00 00 00 00
│  │  └─ PID 0x0D
│  └─ Mode 01（現在データ）
└─ データ長 2バイト
```

**レスポンス例**:
```
03 41 0D 50 00 00 00 00
│  │  │  └─ データ: 80 (km/h)
│  │  └─ PID 0x0D
│  └─ 0x41 = Mode01 + 0x40
└─ データ長 3バイト
```

## OBD2 モード（サービス）一覧

| Mode | 内容 |
|------|------|
| `01` | 現在のセンサデータ（ライブデータ） |
| `02` | フリーズフレームデータ（DTC発生時の記録） |
| `03` | 故障コード（DTC）読み出し |
| `04` | DTC クリア・MIL リセット |
| `09` | 車両情報（VIN 等） |
| `22` | **メーカー独自拡張 PID**（ホンダが多用） |

---

## S07B エンジンについて

N-VAN (JJ1/JJ2) と N-BOX (JF3/JF4) は**同じ S07B エンジン**を搭載している。

```
S07B: 660cc 直3 DOHC VTEC ターボ（ターボ車）/ NA（NA車）
```

これにより、**N-BOX JF3/JF4 での OBD2 実績が N-VAN にそのまま転用できる可能性が高い**。
N-BOX コミュニティでの知見も積極的に参考にする。

---

## 主要 PID 一覧（Mode 01 標準）

| PID | データ | 計算式 | 単位 | N-VAN/N-BOX 状況 |
|-----|--------|--------|------|------------------ |
| `0x00` | サポートPID確認（PID 01〜20） | 32bit ビットマップ | — | — |
| `0x20` | サポートPID確認（PID 21〜40） | 32bit ビットマップ | — | — |
| `0x40` | サポートPID確認（PID 41〜60） | 32bit ビットマップ | — | — |
| `0x60` | サポートPID確認（PID 61〜80） | 32bit ビットマップ | — | — |
| `0x04` | エンジン負荷率 | A × 100 / 255 | % | 取れる可能性大 |
| `0x05` | 冷却水温 | A − 40 | °C | **取れないという報告複数あり** |
| `0x0B` | 吸気管絶対圧（MAP） | A | kPa | **ブースト圧の計算元として使える** ※1 |
| `0x0C` | エンジン回転数 | (A×256 + B) / 4 | rpm | 取れる可能性大 |
| `0x0D` | 車速 | A | km/h | 取れる可能性大 |
| `0x0F` | 吸気温（IAT） | A − 40 | °C | 取れる可能性大 |
| `0x11` | スロットル開度 | A × 100 / 255 | % | 取れる可能性大 |
| `0x2F` | 燃料残量 | A × 100 / 255 | % | 未確認（メータECUが別系統のため怪しい） |
| `0x33` | 大気圧 | A | kPa | 取れる場合あり |
| `0x46` | 外気温 | A − 40 | °C | 未確認 |
| `0x67` | エンジン冷却水温（2センサー） | B − 40 | °C | 一部ホンダ車で代替取得に使用 |
| `0x68` | 吸気温（2センサー） | B − 40 | °C | ターボ後吸気温 候補 |

**※1 ブースト圧の計算:**
```
ブースト圧 (kPa) = MAP (PID 0x0B) − 大気圧 (PID 0x33 or 約101kPa)
ブースト圧 (kgf/cm²) ÷ 98.0665

例: MAP=155kPa, 大気圧=101kPa → ブースト≈0.55kgf/cm²
```

標準PID 0x0Bは**0〜255kPaの範囲**のみ（絶対圧）。
軽ターボのブースト域（大気圧+約60〜80kPa）なら十分カバーできる。

## ホンダ独自 PID（Mode 22）

### Mode 22 の特性

- OBD2 標準の「サポートPID確認」機構が **Mode 22 には存在しない**
- PID は 2バイト（0x0000〜0xFFFF）で非公開
- 調べる手段: ①コミュニティのリバースエンジニアリング結果 ②総当たりスキャン ③HDS等メーカー純正ツール

### 確認済み PID（他ホンダ車での実証あり）

以下は主に **Honda Civic 10th gen（FC1/FK7/FK8、K20C系エンジン）** で Autel AL519 等と照合して確認されたもの。
S07B エンジンの N-VAN/N-BOX への転用は**未確認だが試す価値あり**。

| PID（Mode 22） | データ | 計算式 | 単位 | 確認状況 |
| -------------- | ------ | ------ | ---- | -------- |
| `0x0167` | 冷却水温（ECT） | B − 40 | °C | Civic FC1/FK8 で実測値と一致確認 |
| `0x0168` | 吸気温（IAT、インタークーラー後） | B − 40 | °C | Civic FC1/FK8 で実測値と一致確認 |
| `0x0168` | 吸気温2（IAT2、インタークーラー前） | C − 40 | °C | 同上（同PID、別バイト） |
| `0x2201` | ATF/CVT油温 | A − 40 | °C | 複数ホンダ車で確認 |

> **補足:** `0x0167` / `0x0168` は Mode 22 ではなく **Mode 01 拡張（0x67/0x68）** として解釈される場合もある（ツールによって表記が異なる）。

### 未確認・候補 PID

| PID（Mode 22） | 推定データ | 計算式（推定） | 根拠 |
| -------------- | ---------- | -------------- | ---- |
| `0x1101` | 冷却水温 | A − 40 (°C) | 旧型ホンダ車での報告あり |
| `0x4001` | 冷却水温（別候補） | 要検証 | 一部フォーラムでの言及 |

### ブースト圧について

**Mode 22 でのブースト圧専用 PID は現時点で特定できていない。**
実用上は標準 PID の MAP（0x0B）を使うのが最も確実:

```
ブースト圧 = Mode01 PID 0x0B (MAP絶対圧) − 大気圧(PID 0x33 or 約101kPa)
```

Torque/CarScanner 等では「Boost Calculation Method」を `Prefer MAP` → `Prefer ALT` に切り替えると
より精度が上がる場合がある（ALT = 大気圧センサーとの差分計算）。

### 商用 OBD2 製品の実績（ブースト圧取得確認）

| 製品 | 対応モデル | 取得できるデータ | PID |
| ---- | ---------- | ---------------- | --- |
| BLITZ OBD2-BR1A | 現行ホンダ全般 | 水温・ブースト | **非公開** |
| PIVOT Dual Gauge RS | N-BOX JF3/JF4 S07B(T/C) | 水温・ブースト | **非公開** |
| PIVOT GT GAUGE-60 | N-VAN JJ1/JJ2 S07B(T/C) | 水温 | **非公開** |

これらが動作している = データは取得可能だが、使用 PID は製品の仕様として非公開。

### Mode 22 リクエスト/レスポンス例

```
リクエスト（冷却水温 PID 0x0167）:
03 22 01 67 00 00 00 00
│  │  └──┘ PID 0x0167
│  └─ Mode 22
└─ データ長 3バイト

レスポンス:
05 62 01 67 AA BB 00 00
│  │  └──┘  └─┘
│  │  PID   データバイト
│  └─ 0x62 = Mode22 + 0x40
└─ データ長 5バイト

計算: B − 40 = BB − 40 (°C)
```

## データ取得手順（ステップバイステップ）

### Step 1: CAN 初期化

```cpp
#include "driver/twai.h"

twai_general_config_t g_config = TWAI_GENERAL_CONFIG_DEFAULT(TX_PIN, RX_PIN, TWAI_MODE_NORMAL);
twai_timing_config_t  t_config = TWAI_TIMING_CONFIG_500KBITS(); // N-VAN F-CAN は 500kbps
twai_filter_config_t  f_config = TWAI_FILTER_CONFIG_ACCEPT_ALL();

twai_driver_install(&g_config, &t_config, &f_config);
twai_start();
```

### Step 2: ECU 存在確認（サポートPID問い合わせ）

```
送信: ID=0x7DF, Data=[02 01 00 00 00 00 00 00]
受信: ID=0x7E8, Data=[06 41 00 XX XX XX XX 00]  ← 32bit ビットマップ
```

タイムアウト（200ms）で応答なしなら 250kbps で再試行。

### Step 3: 車速・エンジン負荷を取得（Mode 01）

**車速:**
```
送信: [02 01 0D 00 00 00 00 00]
受信: [03 41 0D AA 00 00 00 00]  → AA (km/h)
```

**エンジン負荷率:**
```
送信: [02 01 04 00 00 00 00 00]
受信: [03 41 04 AA 00 00 00 00]  → AA × 100 / 255 (%)
```

**エンジン回転数:**
```
送信: [02 01 0C 00 00 00 00 00]
受信: [04 41 0C AA BB 00 00 00]  → (AA×256 + BB) / 4 (rpm)
```

### Step 4: 故障コード（DTC）取得（Mode 03）

```
送信: [01 03 00 00 00 00 00 00]
受信: [06 43 NN A1 B1 A2 B2 00]
         │  └─ DTC 件数 NN
         └─ Mode 03 + 0x40
```

**DTC デコード（2バイト → コード）:**

```
バイト1の上位2bit:
  00 → P (Powertrain)
  01 → C (Chassis)
  10 → B (Body)
  11 → U (Network)

例: [01 43] → P0143
    [C1 52] → C1152
```

P1xxx 以降はメーカー独自コードのため、ホンダのサービスマニュアル参照が必要。

### Step 5: 水温を Mode 22 で探索（N-VAN 固有の対処）

標準 PID 0x05 で取れない場合:

```
送信: [03 22 11 01 00 00 00 00]  ← PID 0x1101 を試す
受信: [03 62 11 01 AA 00 00 00]  ← 0x62 = Mode22 + 0x40

計算: AA − 40 (°C)
```

応答がなければ `0x4001` 等、他の候補を順番に試す。

## 実装上の注意点

| ポイント | 内容 |
|---------|------|
| リクエスト間隔 | 最低 50〜100ms 空ける。短すぎると ECU が無視する |
| タイムアウト | 200ms 待って応答なしなら再送（最大2〜3回） |
| 電圧レベル | OBD2 電源は 12V 系。ESP32 は 3.3V → 分圧・レギュレータ必須 |
| CAN トランシーバ | TJA1050 / TJA1051 等が必要（ESP32S3 内蔵コントローラは使えるが物理層は外付け） |
| マルチフレーム | DTC が多い場合は ISO 15765-4 のフロー制御フレームが必要 |
| イグニッション | エンジン停止でも IG-ON なら多くの PID は返答する |

## N-VAN 固有の既知の制限

- 標準 OBD2 PID 0x05（冷却水温）が返ってこないという報告が複数
- PIVOT の N-VAN 対応 OBD2 水温計は製品として存在する → 独自 PID で取っている可能性大
- N-VAN 専用の完全な PID リスト・DBC ファイルは **現時点で非公開・未公開**

## CAN バスの物理的仕組み

### バス型トポロジー

CAN は全ノードが同じ2線（CAN H / CAN L）に並列接続されている。
バスに乗った信号は**全ノードが同時に受信する**。

```
エンジンECU ──┬── CAN H ──┬── AT-ECU ──┬── ABS-ECU ──┬── OBD2コネクタ
              │            │             │              │
             終端          │            終端      ESP32+トランシーバ
             抵抗        CAN L          抵抗
```

### LISTEN_ONLY が成立する理由

CAN フレームには **ACK スロット**がある。送信側がここを Recessive(1) にして、受信した誰かが Dominant(0) に引き下げることで「受け取った」を伝える。

```
通常モード:    ECU 送信 → ESP32 受信 → ESP32 が ACK 返す
LISTEN_ONLY: ECU 送信 → ESP32 受信 → ACK 返さない（黙って受信のみ）
```

ACK を返すノードが**他に1台でもいれば**バスは正常に動く。
ESP32 が黙っていても他の ECU が ACK を返すので、車両の動作には一切影響しない。

### 「メーターに映るデータは傍受できる」の原則

ECU はリクエストとは無関係に、制御のために CAN バスへ定期的にデータを**送り続けている**。
メーターに表示するためにはエンジン ECU がメーター ECU へデータを送る必要があり、
そのフレームが ESP32 にも届く。

```
エンジンECU ──── CAN バス ──── メーターECU
                    │
                  ESP32 ← 横取り（傍受）
```

**メーターに表示されているデータは必ず CAN バスを流れている。**
逆に、メーターに表示されないデータ（ブースト圧等）は流れていない可能性もあるが、
ECU が内部制御のために他の ECU へ送っていれば傍受できる。

## CAN スニッフィング

### OBD2 リクエストとの違い

| | OBD2 リクエスト（アクティブ） | スニッフィング（パッシブ） |
|--|-------------------------------|--------------------------|
| 方向 | プル型（聞きに行く） | プッシュ型（流れてくる） |
| ECU への影響 | リクエスト負荷あり | 一切なし |
| 取得頻度 | リクエスト間隔に依存 | ECU の送信間隔（数ms〜数十ms） |
| 取得できるデータ | OBD2 として公開されているもの | ECU が使っている全データ |
| 解析難易度 | 低（PID → 値の対応が明確） | 高（バイト位置・計算式を自分で特定） |

両者は**同じバス上で同時に動作可能**。TWAI_MODE_NORMAL に戻せばリクエストを送りながらスニッフィングもできる。

### ESP32 (TWAI) での実装

```cpp
#include "driver/twai.h"

twai_general_config_t g_config = TWAI_GENERAL_CONFIG_DEFAULT(
    TX_PIN, RX_PIN,
    TWAI_MODE_LISTEN_ONLY  // バスに影響を与えない
);
twai_timing_config_t  t_config = TWAI_TIMING_CONFIG_500KBITS(); // F-CAN
twai_filter_config_t  f_config = TWAI_FILTER_CONFIG_ACCEPT_ALL();

twai_driver_install(&g_config, &t_config, &f_config);
twai_start();

twai_message_t msg;
while (true) {
    if (twai_receive(&msg, pdMS_TO_TICKS(100)) == ESP_OK) {
        printf("%lu %03X", millis(), msg.identifier);
        for (int i = 0; i < msg.data_length_code; i++)
            printf(" %02X", msg.data[i]);
        printf("\n");
    }
}
```

SavvyCAN 等の PC ツールで読む場合は SLCAN フォーマットで出力する:

```cpp
// SLCAN フォーマット: t[ID][DLC][DATA]\r
printf("t%03X%d", msg.identifier, msg.data_length_code);
for (int i = 0; i < msg.data_length_code; i++) printf("%02X", msg.data[i]);
printf("\r");
```

### キャプチャデータの解析手順

**Step 1: フレーム種別を把握する**

```
ID      送信間隔  用途の見当
0x1xx    10ms    高頻度 → 回転数・スロットル・ブースト系
0x2xx    10ms    高頻度 → 車速・ブレーキ系
0x4xx   100ms    低頻度 → 水温・油温系（変化が遅い）
```

送信間隔が短い = リアルタイム制御データ（回転数・車速・ブースト等）
送信間隔が長い = ゆっくり変化するデータ（水温・外気温等）

**Step 2: 変化するバイトを特定する**

同じ ID のフレームを時系列で並べて差分を見る:

```
時刻   ID    B0  B1  B2  B3  B4  B5  B6  B7
1000  0x158  00  00  1A  F8  00  00  00  00
1010  0x158  00  00  1B  04  00  00  00  00
1020  0x158  00  00  1B  10  00  00  00  00
                        ↑↑↑ B2, B3 が変化している
```

**Step 3: 既知データとの照合**

標準 OBD2 で取った値と突き合わせて意味を特定する:

```
Mode01 0x0C で RPM = 1726rpm 取得済み
→ OBD2 標準の式: (A×256 + B) / 4

0x158 の B2B3 = 0x1AF8
→ (0x1A×256 + 0xF8) / 4 = 1726rpm ← 一致！
→ 0x158 の B2〜B3 = エンジン回転数と確定
```

**Step 4: 状態変化との相関で特定する**

| 操作 | 変化するバイト = 候補 |
|------|----------------------|
| アクセルを踏む | 回転数・MAP・ブースト・トルク |
| 暖機（水温上昇） | 水温・油温 |
| 走行 | 車速・ホイールスピード |
| ブレーキ | ブレーキ圧 |
| エアコン ON/OFF | フラグビット |
| クルーズコントロール ON/OFF | フラグビット（1ビット単位で変化） |

### F-CAN / B-CAN とデータの所在

```
F-CAN (500kbps) → パワートレイン系
  ・エンジン回転数・車速・スロットル
  ・ブースト圧（ECU 間制御に使用している可能性）
  ・クルーズ制御状態（制御結果）
  ・水温・吸気温

B-CAN (125kbps) → ボディ系
  ・ドア開閉・ライト・ウィンカー
  ・クルーズコントロールスイッチ入力 ← スイッチの生信号はこちら
  ・エアコン設定

OBD2 コネクタから直接見えるのは F-CAN のみ。
B-CAN を見るには専用の接続が別途必要。
```

クルーズコントロールの場合、スイッチ信号（B-CAN）より「制御中フラグ」（F-CAN）の方が取りやすい。フラグは1ビットの変化なので比較的特定しやすい。

## Mode 22 PID 探索方法論

### NRC（ネガティブレスポンスコード）を区別する

総当たりスキャン時に応答の有無だけでなく **NRC を区別**することで候補を絞れる:

```
応答パターン:
  05 62 XX XX AA BB ...  → 正常（データあり）
  03 7F 22 31 ...        → NRC 0x31: requestOutOfRange（PID 存在しない）
  03 7F 22 22 ...        → NRC 0x22: conditionsNotCorrect（存在するが今は読めない）
  03 7F 22 33 ...        → NRC 0x33: securityAccessDenied（認証が必要）
  タイムアウト           → ECU がこのモードを未実装
```

`0x22` や `0x33` が返ってきた PID は**存在はする**ので特に注目。

### 公式リファレンスへのアクセス

| 手段 | 内容 | 備考 |
|------|------|------|
| **Honda techinfo** | ディーラー向けサービスマニュアル | 72時間 $10 程度で一般契約可能 |
| **i-HDS / HDS** | Honda 純正診断ソフト | J2534 対応アダプタが別途必要 |
| **Hondata** | K20C 系チューニングツール | 一部 PID が公開されている |

Honda techinfo のサービスマニュアルには「データリスト」項目が記載されており、HDS で表示できる項目名と単位が確認できる（PID 番号そのものは非公開の場合もある）。

## 推奨調査アプローチ

```
1. 標準 OBD2（Mode 01）から確認
   └─ PID 0x00/0x20/0x40 でサポート確認
   └─ 車速・回転数・スロットル・MAP(ブースト元) を取得

2. 水温が必要なら Mode 22 を試す
   └─ 0x0167（Civic 実証済み）→ 0x1101 → 0x4001 の順

3. CAN スニッフィングで全フレームを記録・分析
   └─ TWAI_MODE_LISTEN_ONLY で全フレームをキャプチャ
   └─ 既知データ（OBD2標準値）と照合して ID・バイト位置を特定
   └─ 状態変化（アクセル・暖機・クルーズ ON/OFF）と相関分析

4. Mode 22 総当たりスキャン（必要なら）
   └─ NRC 0x22/0x33 が返る PID を重点的に調査

5. 公式情報が欲しければ Honda techinfo を一時契約
```
