# Claude Code 使い方のコツ

## 1. プロンプトの書き方

「何を・どこを・どうやって」を具体的に伝える。曖昧だと探索にコンテキストを浪費する。

| 悪い例 | 良い例 |
|--------|--------|
| 「バグ直して」 | 「セッションタイムアウト後にログイン失敗する。`src/auth/` のトークンリフレッシュを確認して、再現テストを書いてから修正して」 |
| 「テスト追加して」 | 「`foo.py` にログアウト時のエッジケーステストを追加。モックは使わない」 |
| 「ダッシュボード改善して」 | 「[スクリーンショット貼り付け] このデザインを実装して。結果のスクリーンショットを撮って比較して」 |

- 1プロンプト1タスク。「AとBとCやって」より1つずつ頼む方がトークン効率が良い

## 2. コンテキスト管理（最重要）

コンテキストウィンドウは有限リソース。ほぼ全てのコツはここに帰結する。

- **`/clear`** — タスクが変わったら即リセット。無関係な情報が残ると精度が落ちる
- **2回修正してもダメなら `/clear`** — 学んだことを反映した良いプロンプトで再開する方が速い
- **`/compact <焦点>`** — 長い会話の途中で手動実行。`/compact 〇〇に集中して` で焦点を指定できる
- **`@ファイル名`** — プロンプトに書くとそのファイルを直接参照させられる。画像のドラッグ＆ドロップも可能

## 3. 検証手段を与える（効果最大）

Claude に「自分の出力を確認する手段」を渡すと品質が跳ね上がる。

- 「テストを書いてから実装して」
- 「スクリーンショットを撮って比較して」
- 「`pio run` でビルドエラーがないか確認して」

検証なしだと「それっぽいが動かない」コードが生まれやすい。

## 4. 大きなタスクの4フェーズ

1. **探索**: 「まず関連コードを読んで現状を把握して」
2. **計画**: 「実装方針を立てて」（Plan Mode が自動で入ることもある）
3. **実装**: 承認後にコーディング
4. **コミット**: `/commit`

簡単なタスク（タイポ修正、ログ追加等）は直接実行でOK。

## 5. サブエージェントの活用

- 「自分で必要なコンテキストを取ってきて」と指示すると、サブエージェントに調査を委任してメインのコンテキストを汚さない
- 大規模な調査ほど効果的

## 6. インタビュー手法

大きな機能の場合:

> 「この機能を実装するにあたって、私に質問して仕様を固めて」

自分が考えていなかったエッジケースやトレードオフを洗い出せる。

## 7. CLAUDE.md の育て方

- Claude が間違えたら CLAUDE.md に防止策を追記する。これが最も確実な学習ループ
- 指示は **150個以下** に抑える — 増えるほど全体の遵守率が一様に下がる
- 各行について「これを削除したら Claude は間違えるか？」と自問し、No なら削除する
- コードスタイル（インデント、クォート等）は Linter/Formatter に任せる。CLAUDE.md に書かない

## 8. よくある失敗パターン

| パターン | 症状 | 回避策 |
|----------|------|--------|
| キッチンシンクセッション | 1つのセッションで無関係なタスクを混在 | `/clear` でタスク間をリセット |
| 修正の繰り返し | 2回以上修正してもダメ | `/clear` して良いプロンプトで再開 |
| CLAUDE.md の肥大化 | 重要なルールが無視される | 不要な記述を削除。Hook 化できるものは Hook へ |
| 検証なしの信頼 | エッジケース未対応の実装 | テスト・スクリプトで検証させる |
| 無限の探索 | スコープなしの調査で大量のファイルを読み込み | サブエージェントに委任、または調査範囲を限定 |
